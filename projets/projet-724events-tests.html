Résumé du projet

QA Tester dans une agence web développant le site de 724events, agence événementielle. Le chef de projet Jean-Baptiste confie la mission de lancer la campagne de tests sur la première version du site déjà livrée par l'équipe dev.

Ce projet va plus loin que le précédent : en plus du cahier de recette et des tests exploratoires, il faut analyser en profondeur chaque anomalie détectée. Utilisation d'outils techniques : logs console navigateur, Swagger pour tester l'API indépendamment de l'interface, et Jira pour créer des tickets d'anomalies détaillés.

  • Environnement Docker : accès au site (localhost:3000) et Swagger API (localhost:8080/api-docs), images Docker préparées pour démarrage rapide.
  • Instance Jira : création compte Jira gratuit, board par défaut pour création d'anomalies, export des tickets en fin de projet.
  • Cahier de recette Notion : lecture complète, exécution des tests, notation des résultats et identification des anomalies.
  • Analyse approfondie : pour chaque bug - vérifier logs console, tester API avec Swagger si backend impliqué, déterminer niveau de criticité (mineur/majeur/critique).
  • Tickets Jira : création ticket par anomalie avec template (description, étapes reproduction, criticité, screenshots), positionnement en 1ère colonne du board.
  • Tests exploratoires : 2 sessions (20 min données + 30 min tests libres), minimum 2 anomalies à trouver, respect du temps alloué dans la stratégie.
  • Mise à jour cahier : ajouter les anomalies exploratoires au cahier Notion, noter numéro ticket Jira en commentaire.
  • Bilan de campagne : rédaction finale avec objectifs stratégie, résultats exécution, liste anomalies et décision GO/NO GO.

Analyse technique

Méthodologie d'analyse des anomalies

Pour chaque bug détecté, décider si analyse approfondie nécessaire :

  1. Anomalie simple (wording, image manquante, front pur) → créer ticket Jira direct
  2. Backend suspecté → approfondir l'analyse :
    • Analyser logs console navigateur
    • Utiliser Swagger pour tester API indépendamment
    • Tester chaque route méticuleusement
    • Exemple : /events/{id} nécessite récupérer id via /events/
  3. Ticket Jira : template avec description, étapes, criticité (mineur/majeur/critique), screenshots

  • Décision GO/NO GO selon objectifs
